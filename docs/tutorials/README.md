# ArchHub tutorials

Step-by-step walkthroughs of the workflows ArchHub is best at. Each page
is a recipe that has been run end-to-end against a real project — either
written by a human or auto-minted from a successful trace by the
reflexion worker.

Every tutorial ends with a **Replay this tutorial** button. The button
fires the same flow against a fresh session and verifies it still works.
A tutorial whose replay fails is automatically flagged stale — you will
not waste your time on a recipe that no longer matches the app.

## Where these come from

- **Seed tutorials** are hand-written for the most-asked workflows.
- **Auto-minted tutorials** are produced by the reflexion worker the
  moment a user finishes a flow successfully — the same trace that mints
  the skill also writes a tutorial. See
  `personal-brain-mcp/src/personal_brain/reflexion.py::extract_tutorial_draft`.

Tutorials carry the same scope as the skill they mirror — your firm's
private tutorials sync only across your firm's seats.

## How to add a tutorial

Two paths:

1. **Auto-mint.** Run the workflow end-to-end. The reflexion worker
   notices the success, drafts the tutorial, and submits it for review.
   You approve it from the docs portal.
2. **Hand-write.** Copy `docs/_templates/tutorial.md`, fill the
   frontmatter + sections, drop it in `docs/tutorials/<slug>.md`. The
   index above is regenerated by the docs build step on every commit.

## Freshness signals

Each tutorial declares its `source_paths` in YAML frontmatter. When code
under one of those paths changes, the docs build flags the tutorial as
**potentially stale**. The next nightly replay either re-verifies the
page or marks it stale until someone re-shoots it.
