Composer · Library-first · Multi-LLM enforcement

Direction X with the founder's two refinement mandates baked in. Same prototype shows: model router (5 providers), library-first agent behaviour (search-before-create), modularity validator, user-always-in-control surfaces (library always browsable, canvas editable, Plan/Auto/YOLO modes).

The key insight on multi-LLM enforcement. Anthropic's strict:true is one tool. ArchHub talks to many models. The enforcement must hold for all — so the bulletproof layer is the router-level pre-execute gate, not a per-SDK flag. The per-SDK strict mode is added defence-in-depth where each provider supports it. The system prompt is the same across providers.

The three-layer enforcement

Same constraint, three layers — each provider hits as many as it supports. The router-level gate is the guaranteed floor.

LAYER 1 · SYSTEM PROMPT (works for every model)
"Before composing a new node, you MUST call library.search with the intent + input/output schemas. If any match scores ≥ 0.75, use the existing node. Only call library.create_node_type when search returned no usable match. Every new node MUST declare typed inputs, typed outputs, parameterised config_schema, description, and examples. The validator rejects non-modular specs."
▼
LAYER 2 · PROVIDER STRICT MODE (where supported)
Anthropic: tool_choice:{type:"any"} + strict:true on the tool input schema · forbids hallucinated keys. OpenAI: strict:true on the function JSON schema (post-2024 structured-outputs API). Gemini: tool_config: { function_calling_config: { mode: "ANY" } }. Ollama / LM Studio: defence-in-depth via prompt + post-validation only (most local models don't have a strict-mode wire format).
▼
LAYER 3 · ROUTER PRE-EXECUTE GATE (bulletproof, model-agnostic)
Every tool call passes through ArchHub's router before it executes. The router maintains per-turn state: which tools were called this turn? If library.create_node_type arrives BEFORE library.search on the same turn → REJECTED with a typed tool_result error: { error: "library.search must precede library.create_node_type", retry: "call library.search first" }. The agent re-plans. This works whether the LLM is Anthropic, OpenAI, Gemini, llama3.1, or anything else — the gate is in OUR code, not the provider's.
▼
LAYER 4 · SPEC VALIDATOR (server-side, language-agnostic)
library.create_node_type(spec) validates the spec for modularity: typed inputs ≥ 1, typed outputs ≥ 1, config_schema non-trivial (rejects bare or all-literal configs), description ≥ 20 chars, examples ≥ 1. Any miss → REJECT, return named error, agent rewrites. Same gate for the user's save_as_skill + the agent's mint path.

What this prototype proves

  1. The composer streams tool calls to the canvas across ANY model — the visualisation is identical for Anthropic / OpenAI / Gemini / local — because the canvas reads the router-level event stream, not the SDK directly.
  2. The router gate (layer 3) is the single point that enforces library-first across all models. library.create_node_type without a prior library.search in the same turn = REJECTED, regardless of which LLM emitted the call.
  3. The validator (layer 4) enforces modularity at registration time. A non-modular spec is REJECTED with a typed error explaining what's missing — the agent rewrites.
  4. The library is co-equal with the composer as a user entry point. Cmd-K opens it. Search exists. Saved Skills + AI-minted nodes + core primitives + ai.plan turns all live there together.
  5. Plan / Auto / YOLO modes are visible in the composer footer. Mode change is per-session, never persisted silently.
  6. Reusing existing library nodes is the AGENT'S DEFAULT BEHAVIOUR, visible as REUSED badges on the canvas. New nodes get a NEW · MODULAR badge so the user sees what was minted.
