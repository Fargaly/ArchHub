# Node-Based Application R&D — ArchHub Strategic Review

> Author: Senior product researcher · 2026-05-15. Source corpus: 28
> node-based applications across 6 verticals. Local context:
> `app/web_ui/studio-lm.jsx`, `app/workflows/{graph,runner,subgraph,
> typesystem}.py`, `docs/{NODE_LIBRARY_v2,CANVAS_PLAN}.md`,
> `docs/adr/ADR-003-graph-first-architecture.md`,
> `app/host_detector.py`.

## Executive verdict

**Of all node systems studied, ArchHub should most resemble Dynamo (Revit) sitting on Houdini's cooking model with TouchDesigner's live-preview discipline.** Dynamo because it is the only node system on earth whose users are already AEC professionals — its visual idiom, foreach-over-elements semantics, and Player mode for non-author end-users are the closest analogues to where ArchHub needs to land. Houdini under the hood because lazy/dirty/cached is the only cooking model that survives a 50-node BIM workflow. TouchDesigner's live-thumbnail-on-every-node pattern because architects need to *see* the wall set, not infer it from a port type. **Top three immediate adoptions, ranked by impact × ease**: (1) **node-body value previews** (TouchDesigner / ComfyUI) — the wire engine already caches values; surface them on the node body, not just the hover tooltip. (2) **a Player runtime distinct from the canvas editor** (Dynamo Player) — most architects will *consume* graphs, not author them. (3) **typed `foreach` as a control structure** (Houdini Block Begin/End) — `l_foreach` exists in the library but the runner does not iterate per-element. Everything else can wait six weeks.

## 1. Per-application deep-dive

### 1.1 AEC / parametric

**Grasshopper (Rhino).** Components = typed Python/C# functions; wires carry *data trees* (lists of lists indexed `{0;1;3}`). Push-driven (eager) cook, per-component cache. Weakly typed at the wire (auto-coerces with warning). Subgraph = Cluster (`.ghuser`), file-local. Iteration is implicit via data trees. Distribution via Food4Rhino (~1,500 stock components, ~400 plugins). **The ONE thing**: *geometry IS the preview* — node graph and Rhino model share the viewport. Galapagos = genetic solver. Sources: <https://developer.rhino3d.com/guides/grasshopper/data-trees/>, <https://www.food4rhino.com/en>.

**Dynamo (Revit).** Wraps DesignScript / Python / C#. Lacing modes (Shortest / Longest / Cross Product) are explicit at the node — cleaner than GH. Strict typing — `Element[]` won't drop into a `Point` input. Subgraph = Custom Node (`.dyf`), namespaced. Live preview = per-node text dump. List@level (`list@L1`) for per-input iteration depth — powerful, hard to teach. **The ONE thing**: **Dynamo Player** — a separate executable with no canvas, just saved graphs + inputs + Run. *This is the missing surface in ArchHub.* Sources: <https://primer2.dynamobim.org/>, <https://www.autodesk.com/support/technical/article/caas/sfdcarticles/sfdcarticles/Dynamo-Player.html>.

**Bongo / Karamba3D / Ladybug Tools** — all GH plugins. Karamba = FEM as a node graph. Ladybug = environmental analysis. Pattern worth stealing: Ladybug's *visualizer* components render colored mesh in viewport — output is its own preview. Source: <https://www.ladybug.tools/>.

**Speckle Connectors.** `send` + `receive` + commits + branches (Git-style). Not a cook engine — a transport. **The ONE thing**: **versioned model state, addressable by URL.** Already integrated via `o_spk`/`h_speckle`. Source: <https://speckle.systems/>.

**Bentley GenerativeComponents, Synchro 4D.** GenerativeComponents = MicroStation's GH (~5% AEC market). Synchro 4D's "nodes" are Gantt-tied dataflow — pattern worth stealing later: *time as a first-class canvas axis* (4D scrubber). Defer post-MVP. Source: <https://www.bentley.com/software/synchro-4d/>.

### 1.2 Generative AI / media

**ComfyUI.** Nodes = Python classes; wires strictly typed (LATENT / IMAGE / MODEL / CONDITIONING / CLIP / VAE). Pull-based, queue-driven. Hash-keyed cache. Group Nodes (added 2024) = subgraph collapse to `.json`. Per-node thumbnail rendered directly in node body. Native batching via tensor BATCH dim. **The ONE thing**: **PNG-as-workflow-bundle** — a render is a portable, executable, version-pinned graph; drop the PNG back on the canvas, graph restores. Sources: <https://github.com/comfyanonymous/ComfyUI>, <https://docs.comfy.org/>.

**Cavalry.** Node-based motion graphics. Pattern worth stealing: *attribute drivers* — any numeric value links directly to any other numeric output, no intermediate "math" node. Source: <https://cavalry.scenegroup.co/docs/v2.0/>.

**TouchDesigner.** Five operator families (TOPs/CHOPs/SOPs/MATs/COMPs); wires segregated by family. Lazy + dirty + cached at frame-level — the textbook cook model. Containers (`.tox` toolkits) = hierarchical subgraph. **The ONE thing**: **live previews at 60 FPS** — every operator renders a thumbnail; you *feel* the dataflow. Sources: <https://docs.derivative.ca/Cooking>, <https://learn.derivative.ca/>.

**Notch.** Block-based VFX. Mostly irrelevant to AEC; but the **block-as-template** pattern (a Notch block ships with sliders driving 100s of internal nodes) is exactly what ArchHub Skills do — make sure exposed params stay narrow. Source: <https://www.notch.one/>.

**Houdini SOP/COP/DOP/POP/VOP.** *The* canonical lazy/dirty/cached model — Houdini coined "cook." Strictly typed across families with explicit converter nodes. HDAs (Houdini Digital Assets) = first-class versioned namespaced subgraphs (`fargaly::wall::1.0`). For-each loops are first-class network primitives (`Block Begin / Block End`). **The ONE thing**: **flags** — bypass, lock, cache, freeze, template, displayed, render. Each is a one-click toggle. ArchHub's wire engine has `frozen` only. *The Houdini flag set is the single biggest QoL win for power users.* Sources: <https://www.sidefx.com/docs/houdini/network/cook.html>, <https://www.sidefx.com/docs/houdini/network/flags.html>.

### 1.3 Game engines / simulation

**Unreal Engine Blueprints.** *Exec pins* (white triangles) for control flow + *data pins* (round colored circles) for pull. Strict typing with auto-cast inserts. Subgraph: Functions (call-stacked) + Macros (inline-expanded) + Custom Events + Function Libraries. **The ONE thing**: **exec pins as a separate visual layer.** White wires for control, colored beziers for data. ArchHub's `PortType.EXEC` (`graph.py:72`) exists but the canvas doesn't render exec wires distinctly. Sources: <https://docs.unrealengine.com/5.4/en-US/blueprints-visual-scripting-in-unreal-engine/>, <https://docs.unrealengine.com/5.4/en-US/exec-and-data-pins-in-unreal-engine-blueprints/>.

**Unity Visual Scripting (Bolt).** Two graph types: *Flow Graphs* (Blueprints-style) + *State Graphs* (state machines containing flow graphs). Pattern worth stealing: state graph as a top-level meta-graph orchestrating Skills. Source: <https://docs.unity3d.com/Packages/com.unity.visualscripting@1.9/manual/index.html>.

**Unity Shader Graph + Amplify Shader.** Strictly typed (Vector2/3/4/Float/Texture2D/Matrix). Pattern worth stealing: the *master stack* — the output node has slots for Color/Normal/Metallic in one place. ArchHub could unify `o_pdf`/`o_email`/`o_spk` under a single Publish node. Source: <https://docs.unity3d.com/Packages/com.unity.shadergraph@17.0/manual/index.html>.

**Godot VisualScript (deprecated).** Removed in Godot 4 — unmaintainable for non-trivial logic. *Lesson*: visual scripting cannot replace text scripting; it has to complement it. ArchHub's "drop a Python predicate in `f_pred`" is the right answer.

**Bevy / Stride.** Programmer-facing ECS editors. Not relevant for AEC.

### 1.4 Compositing / shading

**Nuke.** A *script tree* (Python-like `.nk`) and a *node graph* are the same data — *scripting and visual editing are equivalent*. Pull-based, frame-driven. Subgraph = Groups + Gizmos + LiveGroups. **The ONE thing**: **the node graph IS Python.** Type `Read1.file = "shot.exr"` in the script editor; the node updates. Sources: <https://learn.foundry.com/nuke/15.0/content/comp_environment/script_editor/script_editor.html>, <https://learn.foundry.com/nuke/developers/15.0/pythondevguide/>.

**Fusion (DaVinci Resolve).** Like Nuke + the **MultiMerge** node (N-way merge in one node, not a chain of 2-way Merges). *Pattern worth stealing*: ArchHub's `l_merge` should accept N inputs, not 2. Source: <https://www.blackmagicdesign.com/products/davinciresolve/fusion>.

**Substance Designer / Painter.** Node-based texture authoring; *Exposed Parameters* — any internal slider can be promoted to a top-level input. Pattern worth stealing: **expose-as-parameter on right-click**. ArchHub's `subgraph.compose_subgraph` infers facade ports from dangling wires; it should *also* let users right-click an internal slider and promote it. Source: <https://docs.substance3d.com/sddoc/exposing-parameters-186486455.html>.

**Blender Geometry Nodes + Shader Nodes + Compositor.** Three node systems sharing a UI grammar. Geometry Nodes uses *fields* — wires that resolve per-element when cooked. Pattern worth stealing later: fields. Architecturally invasive — defer. Source: <https://docs.blender.org/manual/en/latest/modeling/geometry_nodes/>.

### 1.5 Data / ML / orchestration

**Apache NiFi.** Per-flowfile attributes + content. **Back-pressure** is built-in: downstream queue cap pauses upstream. Pattern worth stealing: NiFi-style queue caps with UI indicator on the wire (queue depth, age). ArchHub's `l_throttle` is a placeholder. Source: <https://nifi.apache.org/docs/nifi-docs/html/user-guide.html>.

**Node-RED.** Msg-passing — every node gets `msg`, mutates `msg.payload`, emits `msg`. **The ONE thing**: **the Debug node** — drop it on any wire, see `msg` in the sidebar. ArchHub's hover tooltip needs hover; a standalone debug node would beat it. Sources: <https://nodered.org/docs/>.

**n8n / Zapier / Make / Power Automate.** Trigger → action chain over JSON payloads. **The ONE thing**: **the catalog** — n8n has 600+ pre-built integrations. ArchHub's 18-host detector is the analogue, but the *actions per host* ratio is the gap (~5x more needed). Source: <https://docs.n8n.io/integrations/builtin/app-nodes/>.

**Airflow / Prefect / Dagster.** Code-first DAG orchestration with **lineage** as a first-class concept. Pattern worth stealing: **asset lineage** — when a Skill produces a wall schedule, record "generated by Skill X v1.4 on date Y from Revit doc Z commit-hash A." Auditable, SOC 2-required. Sources: <https://airflow.apache.org/docs/apache-airflow/stable/concepts/lineage.html>, <https://docs.dagster.io/concepts/assets/asset-materializations>.

**LangFlow / Flowise / LangGraph.** LLM agent orchestration. Memory as a typed wire — a Memory node's output flows into an LLM node's "memory" input. ArchHub's `i_conv` should expose a memory output port. Sources: <https://docs.langflow.org/>, <https://langchain-ai.github.io/langgraph/>.

### 1.6 Design / prototyping

**Origami Studio.** Patch-based UI prototyping. *Switch* patches and *Loop Builder* patches as primitives. Pattern worth stealing: **switch as a first-class node** — `l_switch` exists in ArchHub but is conflated with `l_if`. Origami's switch is multi-valued and *the* pattern for state. Source: <https://origami.design/documentation/>.

**ProtoPie.** Trigger / Response interaction model. Less node-y, more event-y. Mostly irrelevant.

**Figma auto-layout.** Constraint-based, not nodes. Worth stealing nothing for the engine; but the *constraint layout* idea is what ArchHub's auto-layout needs to be — push apart on collision, snap to alignment.

**Cables.gl.** Browser-based real-time visual programming. Same TouchDesigner lessons. Notable for shipping *the canvas inside the browser* — exactly the architecture ArchHub uses (QtWebEngine + JSX). Source: <https://cables.gl/docs/>.

### 1.7 Programming language theory

**Pure Data / Max/MSP.** Message-passing dataflow. Async by default. Message tagging — `[symbol value1 value2 ...]` first symbol routes to right inlet. ArchHub's wire types do this job at design-time; runtime message tagging is overkill. Sources: <https://puredata.info/docs>, <https://docs.cycling74.com/max8/vignettes/messages>.

**LabVIEW.** Dataflow with explicit wires. Rich type system (clusters, arrays, references). *Lesson learned*: LabVIEW shows strict typing + visual programming scales to 10K-node engineering systems. ArchHub's `typesystem.py` is on the right track. Source: <https://www.ni.com/docs/en-US/bundle/labview/page/lvconcepts/dataflow.html>.

**dbt, sqlmesh.** SQL-as-DAG. **Materialisation** as first-class (table / view / incremental / ephemeral). Pattern worth stealing: **materialisation per node.** ArchHub already has `frozen` (= cached). Add `materialised` (= write to disk for inspection) and `ephemeral` (= never cache). Source: <https://docs.getdbt.com/docs/build/materializations>.

## 3. Synthesis

### 3.1 Patterns ArchHub already does well — leave alone

- **Strict, explicit type coercion table** (`app/workflows/typesystem.py:_COERCIONS`). One-way arrows, no auto-coerce. Better than GH, on par with Dynamo/ComfyUI. *Architects rage when wires silently coerce.*
- **Lazy / dirty / cached cooking** (`app/workflows/runner.py:WorkflowRunner.pull`). Houdini-class with per-node hash cache keys.
- **Profound wires with field selectors** (`runner.py:_resolve_field`, `_wrap_field`). `src_field="selection.walls"` is a TouchDesigner CHOP-style dotted-path. Stealth power feature.
- **Edit-time cycle prevention** (`runner.py:would_create_cycle`). Prevents the worst class of runtime crash.
- **Subgraph compose/expand** (`subgraph.py`). Inverse operations, dangling-wire inference. Better than ComfyUI Group Nodes.
- **80-node taxonomy across 10 categories**. Pragmatic — no GH-style 1,500-component bloat.
- **Composer slash-commands as a parser** (`composer_commands.py`). Nuke-class — graph operations expressible as text.
- **Bridge of 115+ `@pyqtSlot`s** keeping JSX state and Python graph in sync via QWebChannel. Cables.gl-class.

### 3.4 Patterns ArchHub should explicitly NOT adopt — and why

- **Grasshopper Data Trees.** Tree paths `{0;1;3}` work for parametric facades; they confuse architects who want a wall schedule. Flat list + `l_foreach` is the right model.
- **Houdini's full 8-flag matrix.** Pick the 3 that matter (`bypass`, `template`, `displayed`). The other 5 are studio-FX hygiene AEC users will never touch.
- **TouchDesigner family-segregation.** ArchHub's wire types are already segregated *enough*. Hard family barriers complicate the canvas without buying anything for AEC.
- **Pure Data / Max message-passing runtime.** Async dispatch is great for audio; for AEC it would obscure cause-and-effect and break lazy/dirty/cached.
- **Unity State Graph as the *primary* surface.** Nodes-of-nodes is a power-user feature. Don't make it the default.
- **ComfyUI's no-sandbox custom node model.** Architects will install random skills from the internet. Sandbox custom nodes (subprocess + capability whitelist) before any marketplace.
- **Blender Geometry Nodes "fields."** Conceptually beautiful, architecturally invasive. Defer to post-v2.0.
- **Houdini-class HDA syntax.** Adopt the *idea* (versioned namespaced Skills) but not the `::` syntax. Use `fargaly/wall@1.0`.
- **Unreal Blueprints as the *only* control flow.** Exec pins are great for *event handlers*; for *data transforms* the dataflow model is correct. Use both — *don't push exec pins into reads/filters.*
- **Per-frame TouchDesigner cooking at 60 FPS.** ArchHub doesn't have a frame loop; cook on edit + cook on Run is correct.
- **Galapagos genetic solver.** Sexy; no pre-MVP demand. Defer indefinitely.
- **n8n-style "every SaaS is a node."** Tempting; the integration tax is brutal — ArchHub's strength is *AEC depth*, not *integration breadth*. Resist the urge to add a Slack node until 50 architects ask.

| Node | Source | Why ArchHub needs it | Suggested id |
|---|---|---|---|
| **Debug / Inspect** | Node-RED Debug | Pin any wire's last-cooked value to right sidebar. Solves "why is this empty?" without hovering. | `l_debug` |
| **For-Each Begin / End pair** | Houdini Block Begin/End | True iteration over list elements with an inner subgraph. Today's `l_foreach` is a label. | `l_foreach_begin`, `l_foreach_end` |

### 3.6 The 3 architectural shifts the wire engine / runner needs

#### Shift 1 — Promote `value_preview` from tooltip to first-class UI surface

Today: `Edge.value_preview` (`graph.py:148`) populated after every cook, surfaced only as hover tooltip in JSX. Tomorrow: every cooked node renders its preview in a 60–80px footer band. Type-dispatched renderers:

- `walls`/`doors`/`elements` → tiny SVG bbox plan
- `dims`/`tags` → text count + sample
- `completion`/`message` → first 200 chars
- `file`/`pdf` → filename + size
- `selection` → element count + types

New dispatch in `studio-lm.jsx` `NodeRenderer`. Backed by existing `PERSISTABLE_TYPES` whitelist (`runner.py:194`) so previews stay bounded.

**Why this matters more than anything else**: Architects don't trust black-box graphs. They want to see the wall set after every step. With per-node previews, they see it on the source node. *This is the single change that makes the canvas feel intelligent.*

#### Shift 2 — Add a flag system + a Player runtime that respects it

Today: only `node.frozen` (`runner.py:422`). Tomorrow: a flag bitmap per node — `frozen | bypass | template | displayed | cache_to_disk`. The *displayed* flag tells the Player runtime "this is what the user wants to see."

Player (`app/web_ui/player.jsx`, ~600 LOC) is parallel to `studio-lm.jsx`. Opens a Skill, lists facade input parameters, lists `displayed`-flagged outputs, runs the workflow, shows displayed outputs as cards. Zero canvas. *This surface gets ArchHub from "10 power users" to "1000 architects."*

#### Shift 3 — Lineage as a first-class side effect of cook

Today: cook returns outputs, leaves no trail. Tomorrow: every `runner.pull` appends to `%LOCALAPPDATA%\ArchHub\lineage\<session>\<run_id>.jsonl`. Each line: `{ts, node_id, type, inputs_hash, output_preview, host_state, duration_ms}`. Queryable: "Show me every run of `wall_schedule` last week."

Required for: SOC 2 audit trail (`docs/SOC2_READINESS.md`), debug ("why did this schedule have 47 walls yesterday and 39 today?"), cost accounting (LLM tokens per node per run), Skill performance ("which Skills are slowest?").

Implementation: `LineageRecorder` injected into `WorkflowRunner.ctx` (~100 LOC), called from `pull` after every successful cook, batch-flushed on a timer.

## 4. Six-week R&D backlog

Each week ~1 engineer with PyQt6 + JSX fluency. Sequential where dependencies exist.

### Week 1 — Node-body live previews

- `studio-lm.jsx` `NodeRenderer` (~line 1990): add 60-px preview footer with per-type renderer. ~200 LOC.
- New `studio-lm.jsx` `previewRendererForType(type, value)` dispatch table. ~150 LOC.
- Bridge: ensure `value_preview` available on every cook (`runner.py:_emit` already does; verify edges).
- Tests: `tests/test_runner_preview.py` — cook each library node, assert preview non-empty.
- **Ship gate**: 100% of node types render *something* in the footer.

## 5. Closing notes

- **The cooking model is not the bottleneck.** ArchHub's `runner.py` already implements lazy/dirty/cached at Houdini parity. The bottleneck is *what happens around the cook*: previews, flags, lineage, Player. All UI-adjacent. All shippable in 6 weeks by one engineer.
- **The category list is right.** 80 nodes / 10 categories matches Dynamo's surface area. Don't expand to GH's 1,500 — the long tail is noise. Add the *5 missing nodes* in §3.5 and stop.
- **The host matrix is the moat.** 18 hosts is more than ComfyUI, Dynamo, GH, and TouchDesigner combined. Defend it — every node should *do something concrete with a host*, not be a generic data-transform.
- **The agent composer (`bridge.agent_step`) is structurally ahead of every competitor.** No node-based application studied has a *graph-mutation LLM* with a typed-tool schema. This is the differentiation. Surface it more aggressively in onboarding.
- **The biggest risk is "looks like ComfyUI but doesn't show me my walls."** Per-node previews fix this in week 1. Ship it first.
